# CLI設計の好み

## サブコマンド構成

- 複数機能 → サブコマンドで提供（独立コマンド乱立を避ける）
- PoC/プロトタイプ段階は独立コマンドも許容
- 子・孫のネストしたサブコマンドを持てること

### `--help` 表示構成

セクション分けして表示:

1. サブコマンド一覧（そのレベルの子）
2. そのサブコマンドレベルのオプション
3. グローバルオプション
4. 利用可能な環境変数（サブコマンドレベル / グローバルレベル）

### 引数パーサ選定

- 言語に応じて上記要件を実現可能なパーサを選択
- 全要件を満たすパーサがなければ妥協も検討するが、自作も厭わない

## 引数なし実行

- 引数・サブコマンドなしで実行時は `--help` を表示
- トップ・子・孫の全サブコマンドで共通

## オプション

- ロングオプションを基本とする（補完前提）
- ショートオプションエイリアス（`-h`, `-c`等）を指示なく追加しない

### boolフラグ

- デフォルト `false`、指定時 `true` が基本
- デフォルト `true` のフラグも許容。反転は `--no-xxxx` 形式
- feature系フラグが多い場合は `--enable-xxx` / `--disable-xxx` パターンも可

### 引数の位置

- オプション引数の位置固定を避ける（メイン引数の後ろにもオプションを置ける）
- メイン引数との曖昧さ対策には `opts... -- args...` の `--` セパレータを用意する

## 補完

- CLIツールはcompletion出力機能を持つべき
- `*.plugin.zsh` で提供可能
- 過不足のないコンテキスト対応補完
- **3 つを同期保守**: 実装 (option parser / 振る舞い) ↔ `--help` テキスト ↔ completion 定義
  - 実装変更時: `--help` と completion を同時に追従
  - `--help` 変更時 (= option 追加 / 削除 / 引数仕様変更): completion 定義も必ず追従
  - completion 追加時: `--help` の出力と齟齬なく
- 片方向だけ追従するアンチパターン (= 「help 更新したけど completion 古いまま」「completion 直したけど help 古い」) は **片面 rule の盲点** ([[self-written-rule-blind-spots]])、3 者の整合は意識的に確認
