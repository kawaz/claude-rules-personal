# 撤退・諦め判断は最後の手段

bug / 不具合に直面したとき、「機能を削る」「仕様の限界として諦める」「設計を覆す」は
**最後の手段**。まず「実装漏れ / config 漏れ / 環境設定」を疑う。

## 疑う順序

1. **実装漏れ**: 設計済みの機能が実装されていない or 一部しか実装されていない
2. **config 漏れ**: 設定値・環境変数・feature flag が抜けている
3. **環境設定**: OS / shell / 周辺ツールの違い
4. **依存ライブラリの挙動差**: バージョン / patch level の違い
5. **(ここまで全て否定された後)** 仕様の限界 / 設計の問題

## How to apply

- bug 報告を受けたら、最初に「設計済み機能が実装されているか」を grep で確認
- 「新規発見した root cause 候補」を即「真の根本原因」と断定しない
- 撤退案を提示する前に、上記 1-4 の各レイヤを潰した記録を残す
- 削除 PR を出すなら、削除理由として 1-4 を否定した根拠を本文に書く

## Why

撤退判断は不可逆。先に簡単に直る原因を見落としたまま機能を削ると、
後から「あれ実は直せた」が判明しても戻すコストが大きい。
