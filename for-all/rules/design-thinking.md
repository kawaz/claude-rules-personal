# 設計思考

## ワークアラウンドフィールド禁止

問題解決のために構造体にフィールドを追加するのは最終手段。

1. **既存の枠組みの組み合わせで解決できないか**を最優先で考える
2. 既存で無理でも、**そのフィールドが本当に正しい場所か**を疑う
3. 表示用メタデータのために実装を歪めるのは本末転倒

### チェックリスト

新しいフィールドや型を追加する前に:

- [ ] 既存の型・関数の組み合わせで表現できないか？
- [ ] クロージャで状態をキャプチャすれば不要にならないか？
- [ ] そのフィールドは本当にその構造体の責務か？
- [ ] ヘルプ表示等の「従」の都合で「主」のコア設計を歪めていないか？
- [ ] 将来の仮定的要件のために今の複雑さを増やしていないか？

## 思考設定

- システム設計: ドメインモデリングからアーキテクチャを考える
- 計画立案: ゴールから逆算して計画する
- 説明方法: 具体例を先に示してから抽象的な概念を説明する
- 解決アプローチ: 問題を小さく分解して段階的に取り組む
