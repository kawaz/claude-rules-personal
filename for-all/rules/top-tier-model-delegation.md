# 最上位 tier モデル使用時のモデル選定

メインセッションが Anthropic 提供の **最上位コスト tier** (= `/model` で
選択可能な最高 tier、現時点の代表例は Fable) で動作している場合、以下の
分担で作業する。メインが中位 tier (Opus / Sonnet 等) なら本ルールは適用せず、
[[work-principles]] の「サブエージェント活用」の中位 tier 節に従う。

自 tier の判定は system prompt 中のモデル名 (例: `claude-fable-*` = 最上位、
`claude-opus-*` / `claude-sonnet-*` = 中位) で行う。判定不能時は中位 tier
扱いを safe default にする (= 最上位扱いで高コスト誤動作するより安全側)。

原則: **コスト最適化は実作業 (量産系) にのみ適用し、品質判定 (レビュー・監査) には適用しない**。
品質判定を自分より劣る tier に委譲するのは本末転倒 (= 劣るモデルが見逃す問題を見つけるのが目的なのに、
見逃す側に判定させることになる)。

## 分担

- **メイン (最上位 tier)**: 設計・タスク分解・サブエージェント成果の監査・統合・ユーザ対話に専念
- **本気レビュー・設計監査・最終品質判定**: **最上位 tier のサブエージェント** に出す
  - リリース前レビュー、設計文書の批評、「辛口で」「本質的な指摘を」と言われた場面が該当
  - background 実行可
  - 観点分割並列 vs 1 本全方位: 暫定優先案は **1 本全方位** (= N=1 ながら実測あり:
    claude-plugin-reference レビューで Fable 1 本 148k tokens が Opus 3 本 322k tokens
    の主要指摘をカバーしつつ Critical 級を 5 件追加した観測)。サンプル追加で覆る可能性
    があるため、必要に応じて両方を試して `docs/findings/` に積む ([[empirical-verification]]
    のサンプル数原則と整合)
- **スクリーニング的レビュー (定型チェック・lint 的網羅・大量ファイルの粗探し)**: 中位 tier 並列で可
- **実作業 (実装・検証・調査・doc 執筆)**: 中位 tier のサブエージェントに切り出す
  - 検証設計・実装判断が細かい作業 → Opus 等
  - CLI 観測・定型実装・doc 整形・記録 → Sonnet 等
- メイン直の例外条件は [[work-principles]] の「メインが直接行ってよい作業」を参照
  (= 軽微編集 / 設計と実装が不可分な高難度箇所、判定基準も同節に集約)

## Agent / Workflow 起動時の必須事項

**`model` を必ず明示する**。未指定のサブエージェントは**メインのモデルを継承**する
(= 最上位 tier で実作業が走り高コスト)。Workflow / Agent tool のいずれの経路でも同様。

model 未指定で起動してしまった場合は、序盤なら止めて指定し直すほうが安い。

専用 agent (`nitpick-reviewer` / `codex:codex-rescue` 等) は各 agent の `model`
フロントマター指定が優先、未指定なら親継承。詳細は [[claude-plugin-reference]] skill。

## Why

- 最上位 tier はサブエージェント分も含めると支配的コストになる。監査・統合はメインの強み、
  量産的な実作業は下位 tier で十分品質が出る
- 「未指定 = メイン継承」は見落としやすく、気づかず全サブエージェントが最上位 tier で走る事故が起きる
- レビュー品質はモデル tier に概ね比例する (体感ベース)。レビューの取りこぼしは
  実装ミスより高くつく (= 信頼して出荷される) ため、ここだけはコストより品質を優先する
