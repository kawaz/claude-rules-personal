# 最上位 tier モデル使用時のモデル選定

メインセッションが最上位コスト tier のモデル (現: Fable 5) で動作している場合、
以下の分担で作業する。メインが Opus / Sonnet 等なら本ルールは適用しない。

原則: **コスト最適化は実作業 (量産系) にのみ適用し、品質判定 (レビュー・監査) には適用しない**。
品質判定を自分より劣る tier に委譲するのは本末転倒 (= 劣るモデルが見逃す問題を見つけるのが目的なのに、
見逃す側に判定させることになる)。

## 分担

- **メイン (Fable)**: 設計・タスク分解・サブエージェント成果の監査・統合・ユーザ対話に専念
- **本気レビュー・設計監査・最終品質判定**: **Fable サブエージェント** (メインと同 tier) に出す
  - リリース前レビュー、設計文書の批評、「辛口で」「本質的な指摘を」と言われた場面が該当
  - background 実行可。観点分割で並列するより **1 本に全方位を見せる**ほうが効く
    (実測: claude-plugin-reference レビューで Fable 1 本 148k tokens が Opus 3 本 322k tokens の
    主要指摘をほぼ単独でカバーした上に、Opus 全滅の Critical 級を 5 件追加した)
- **スクリーニング的レビュー (定型チェック・lint 的網羅・大量ファイルの粗探し)**: Opus 並列で可
- **実作業 (実装・検証・調査・doc 執筆)**: Opus / Sonnet のサブエージェントに切り出す
  - 検証設計・実装判断が細かい作業 → Opus
  - CLI 観測・定型実装・doc 整形・記録 → Sonnet
- 例外: 実装難易度が特に高い箇所 (設計と実装が不可分な箇所) や、委譲のほうが高くつく
  軽微な編集 (数行の修正・小ファイル 1 個) はメインで直接やってよい

## Agent / Workflow 起動時の必須事項

**`model` を必ず明示する**。未指定のサブエージェントは**メインのモデルを継承**する
(= Fable で実作業が走り高コスト)。Workflow script の `agent()` の `opts.model` も同様。

model 未指定で起動してしまった場合は、序盤なら止めて指定し直すほうが安い。

## Why

- Fable はサブエージェント分も含めると支配的コストになる。監査・統合はメインの強み、
  量産的な実作業は下位 tier で十分品質が出る
- 「未指定 = メイン継承」は見落としやすく、気づかず全サブエージェントが Fable で走る事故が起きる
- レビュー品質はモデル tier に強く比例する (上記実測)。レビューの取りこぼしは
  実装ミスより高くつく (= 信頼して出荷される) ため、ここだけはコストより品質を優先する
