# 業務文脈の固有名詞を個人プロジェクトの永続化成果物に書かない

個人プロジェクト (`~/.claude` 環境) での作業時、業務固有の固有名詞・話題カテゴリを
**永続化される成果物** に含めない。具体的な対象は `identifiers-*.md`
(各 overlay リポ) で定義する。

## 永続化される成果物 (要サニタイズ)

- コミットメッセージ・PR タイトル/本文
- docs/, README, journal, handoff 等のリポ内ファイル
- コードコメント・識別子 (変数名・関数名)
- 公開リポ (kawaz/* の public) への push 物は特に厳格

## サニタイズ対象外

- セッション内の会話・思考 (チャット履歴は個人 public に出ない)
- `~/.claude-work` 環境での作業 (業務リポ内)
- 業務リポ自身への commit
- ユーザが明示的に「固有名詞含めてよい」と指示した場合

## 一般化の例

| 業務固有 | 一般化 |
|---|---|
| 具体的な組織名 | 「ある業務先」「業務関連組織」 |
| 業務用 GitHub アカウント名 | 「業務用 GitHub アカウント」 |
| 業務リポ名 | 「業務リポ」「仕事用リポ」 |
| 業務固有プロダクト名 | 「業務プロダクト」 |
| 話題カテゴリ (特定業界・特定領域) | カテゴリ名を伏せて「ある業界」「特定領域」等 |

## 適用フロー

1. **Write/Edit 前**: 書き込み内容に対象単語・カテゴリが含まれていれば、書き込み前に一般化を提案
2. **コミット直前**: staged diff を確認、対象が含まれていれば commit を止めて一般化提案
3. **公開 push 前**: kawaz/* の public リポへの push 前は特に厳格にチェック
4. **判断に迷ったら**: AskUserQuestion で「業務固有名詞が含まれていますが、このまま進めるか / 一般化するか」を提示

## 単語マッチと文脈判定

- **単語リスト** (`identifiers-*.md` の単語リスト) は文字列マッチでよい
- **話題カテゴリ** は単語列挙では弾けない。文脈判定が必要 (例: 「特定業界の業務文脈で書かれているか」)
- 単語が一般名詞と衝突する場合 (例: "party") は文脈判定に切り替え、業務文脈で使われているときのみサニタイズ対象

> **Why**: 個人 public リポへの業務情報流出防止。kawaz/* を public 化した場合や、
> 個人プロジェクト docs を OSS 公開する時に、業務関連情報が含まれないようにするための予防線。
> 固有名詞の具体リストは各 overlay リポ (private) で管理し、本ルール (公開可能) には
> 仕組みだけを書くことで、common 層を public にしても情報漏洩しない構造を維持する。
