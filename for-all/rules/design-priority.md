# 設計の優先順位

設計判断は以下の観点を最優先する:

1. **設計が正しいか**（ドメインモデル・責務分離・抽象化レベルの妥当性）
2. **ベストプラクティスに則っているか**
3. **どちらがより良い設計か**（複数案の比較で本質的に優れている方）

これらを以下の理由で曲げない:

- 「後方互換性のため」
- 「修正が大変」「影響範囲が広い」
- 「過去の経緯がある」「すでに動いている」

## 適用

判断に迷ったら **より正しい設計を選んで全体を直す**。差分の小ささや書き換えコストは
エージェント（自分）が払うコストであり、ユーザに対する理由として持ち出さない。

## チェックリスト

- [ ] 「diff を小さくするため」を理由に妥協していないか
- [ ] 「rename すると影響範囲広い」を理由に命名を放置していないか
- [ ] 「既存 API があるから」を理由に重複/平行 API を増やしていないか
- [ ] 「動いてるから」を理由に設計上の歪みを残していないか
- [ ] 不採用案を選んだとき、理由は「設計上の優位性」になっているか（コストではなく）

## 対極 — 「全体を直す」のスコープ制御

「正しい設計を選んで全体を直す」を無限適用するとスコープ爆発の口実になる
([[self-written-rule-blind-spots]])。歯止め:

- 「全体を直す」の範囲は**現タスクの完了条件** (orchestrate skill Phase 0 の
  3 行アンカー) と照合する。アンカー外に膨らむ正しい設計は **issue 起票して
  現タスクを完了させる**のが正、無断でスコープに繰り込まない
- 本ルールが禁じるのは「コストを理由に**質を**曲げる」こと。「今回やる**範囲を**
  絞る」ことは禁じていない (= 絞った範囲の中では正しい設計で直し切る)

## 適用範囲

- 個人プロジェクト（自分管理）: 強く適用
- 公共リポジトリへのコントリビューション: 受け入れ側スタイルに従う（`public-repo-contribution.md`）
- 外部公開 API / プロトコル: 後方互換が「設計上の責務」になる場合のみ別判断
