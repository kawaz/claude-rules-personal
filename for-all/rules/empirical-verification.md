# 経験的検証 (実機マトリクス主義)

判断の根拠にする事実は **実機 / 観測道具で確認**する。推測 / 流し読みは禁止。

## マトリクス検証

「期待 vs 実態」の表を、関連する全組合せで埋める。1 ケースの観察で結論を出さない。

- **サンプル数原則**: 最低 3 種類の category で検証する (例: 異なる app / env / config / OS バージョン等の categorical 軸を最低 3 つ)
- 1 サンプルだけで「こういう仕様だ」と結論しない
- 反例が出る軸を能動的に探す

## 観測道具を使う

推測ベースの「たぶんこう動く」を禁止し、実体を観測する道具を使う:

- プロセス / シグナル: `ps`, `lsof`, `dtrace`, `strace`
- 端末: `stty`, `script`, `tmux capture-pane`
- ネットワーク: `tcpdump`, `curl -v`, browser devtools
- snapshot / log / diff の取得を恒常的に
- 「動いているはず」「こう書いてあるから」で済ませない

## How to apply

- 新しい現象に気づいたら、まず観測してから解釈する
- 「1 つの app で動いた」では仕様の主張をしない (= サンプル数 1)
- ドキュメント記載と実機挙動が食い違ったら、実機を優先して記録する
- 検証マトリクスは `docs/findings/` 等に表形式で残す

## 検証しきった後の振る舞い

裏取りの責任は Claude 側にある。ユーザーに証拠の目視確認を肩代わりさせない。

- 自分で観測道具を使って確認**できる**ことは、確認しきってから話す。
  「私の証言だけでは不安なので確認してほしい」とユーザーに検証を投げて
  止まらない。それは裏取りの放棄であって慎重さではない。
- 自分で確認した事実は、確信を持って述べる。確認済みのことに過剰な留保を
  付けない (裏取りを徹底した結果、確信の表明まで避けるのは本末転倒)。
- ユーザーに委ねてよいのは、Claude が**確認不能**なことだけ
  (実機アクセス不可・権限がない・実行の副作用が大きく承認が要る等)。
  その場合も「何が確認できて、何が確認できないか」を明示し、丸ごと
  「確認してください」と投げない。
- 判断基準: その確認は自分で `ls`/`grep`/コマンド一発でできるか?
  できるなら自分でやる。それで初めて確信を持って結論できる。

## 対極 — 検証を省く / 軽くする判断 (= ROI gate)

「検証しろ」だけだと過剰になる ([[self-written-rule-blind-spots]])。
対極の判断も持つ:

- **1 サンプルで足りる場面**: 決定的 / 冪等 / 読み取り専用の動作確認、
  公式一次資料の記述と実機 1 点の一致確認。サンプル数 3 原則は「仕様を
  主張する」時の基準であって、全確認行為への一律要求ではない
- **検証コスト > 手戻りコストなら省略してよい**。ただし黙って省略せず
  **「未検証」と明記**する (= 蓋をするのではなく可視化する、
  [[test-failure-no-tampering]] と同じ思想)。可逆で局所な変更ほど省略側、
  不可逆・公開・報告の根拠になるものほど検証側に倒す
- 省略の判断に迷ったら検証する (= safety net は検証側)

## Why

実装 / OS / 端末 / library の挙動は文書と乖離していることが多い。
推測ベースの判断は再現性も裏取り可能性もない。
