# 調査・検証の記録ルール

## 原則

- 検証結果・調査結果は都度ファイルに記録する（コンテキストやメモリではなく）
- 記録先: `docs/findings/` 等、プロジェクト内のナレッジディレクトリ
- 記録はバックグラウンドのサブエージェントに委譲し、ユーザーとの会話をブロックしない

## ファイル構成

```
# タイトル
## 判明した事実
（確定事実のみ。推測や仮説は含めない。これだけ読めば結論がわかる）
## 実用的な示唆 / ベストプラクティス
## 検証の詳細
（各テストの結果テーブルと考察。後述として残す）
```

命名: プロジェクト固有のルールがなければ `YYYY-MM-DD-{title}.md`

## 記録委譲ワークフロー

### サイクル

1. ユーザーが結果・発見を報告
2. メインは**即座に分析・応答**（次の提案、質問など）
3. **同時に**バックグラウンドサブエージェントを起動して記録を委譲
4. 完了通知で軽く報告（「記録完了」程度）、記録完了を待たずに会話継続

### サブエージェントへのプロンプトパターン

コンテキストは引き継がれない。必要な情報を**全て**プロンプトに含める:

```
ファイル {path} を Read して以下を更新:

1. 「## 検証の詳細」セクション末尾に追加:
### {テスト名}
{結果テーブル・考察}

2. 新たに確定した事実があれば「## 判明した事実」も更新:
- {確定事実}
```

### 注意点

- 同一ファイルへの同時書き込みを避ける（前の記録エージェント完了後に次を起動）
- ファイル新規作成は初回のみフォアグラウンドで構造を作り、以降はバックグラウンド追記
- テーマが変わったらファイルを分割（1ファイル1テーマ）
- 「判明した事実」と「検証の詳細」の**両方を同期更新**するよう毎回指示に含める
