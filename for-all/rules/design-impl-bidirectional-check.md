# 設計と実装の双方向整合性チェック

設計文書 (DR / spec / DESIGN.md 等) と実装コードは、**双方向**で整合性を確認する。
片方向だけだと「設計済みなのに未実装」が長期間放置される。

## 双方向の中身

- **A 方向 (実装→設計)**: 新規介入を入れる前に「既存 DR で justify されているか」を確認
- **B 方向 (設計→実装)**: 設計文書を読んだら「その機能の実装エビデンス」を grep / test で確認

## How to apply

- 新規 PR / change を作る前に A 方向: 関係する DR を読み、自分の変更が既存方針と整合するか確認
- DR / spec を読み終わったら B 方向: その DR が「実装済み」を主張する範囲を grep で裏取り
- self-check ルールや design-priority ガードを作るときは、必ず両方向を同時に組み込む
- A 方向だけの check は危険信号 — 「未実装の放置」を発見できない

## Why

設計文書は判断記録であって実装の証明ではない。「書いてあるはず → 動いているはず」の
推定は実装漏れを覆い隠す。grep / test で実体を確認することで初めて整合が取れる。
