# テストファースト開発

機能追加・bug fix では **t_wada の TDD** を実践する:

1. **RED**: 仕様の輪郭を明確化する失敗テストを書く
2. **GREEN**: 最小実装でテストを通す
3. **REFACTOR**: テストが通った状態を保ったまま改善

## RED の目的: 仕様の輪郭を明確化する

「ただ失敗するだけのテスト」を量産しない。テストケースの目的は **こうなるべき
仕様 (ゴール) を表現する** こと。惰性で穴だらけのテストを並べるのは最悪
(= 通っても仕様が担保されない、後で書き直しが必要)。

## テストケース設計の観点

最低限すべて意識し、該当しない観点は「該当なし」と明示的に判断する:

- **正常系**: 主要なユースケースが期待通り動く (= ゴールの達成)
- **異常系**: 不正入力 / 失敗パスで適切に失敗する (= 黙って成功 / 黙って成功風で
  済まさない、エラー型・メッセージも仕様)
- **境界ケース**: off-by-one / 最小値・最大値 / 空入力 / null・undefined /
  長さ上限・桁上限 / 0・負数・小数の取り扱い
- **エッジケース**: タイムゾーン / ロケール / unicode (surrogate pair・絵文字・
  結合文字) / 浮動小数 precision / 桁あふれ / 環境差分
- **複数条件 (デシジョンテーブル)**: 条件の組合せを表で網羅。independent な
  n 条件で 2^n の組合せが出るので、必要組合せを精選 (ペア網羅 / 危険なペア優先)。
  「全部 OR / 全部 AND」だけで済ます手抜きを避ける
- **同値分割**: 入力空間を「同じ振る舞いになる集合」に分け、各集合から代表値を
  1 つずつ選ぶ。境界ケースと組み合わせて入力空間を覆う
- **状態遷移**: ステートマシン的振る舞いがあれば、各遷移 / 不正遷移 / invariant
  を確認
- **副作用 / べき等性**: 同じ操作を繰返したとき同じ結果か、副作用の有無、
  ロールバックの可能性
- **並行性 / レース**: 並列実行で正しく動くか (共有状態 / 競合 / デッドロック /
  順序保証)
- **回帰**: 過去の bug 報告 / 修正済みケースが再発しないか (= 修正と同時に
  そのケースのテストを残す)
- **性能境界** (該当時): 大入力でタイムアウト / メモリ越えしないか、
  許容ライン以下で完了するか
- **セキュリティ境界** (該当時): injection / 認可越境 / 入力エスケープ /
  権限昇格パスの検証

複数条件が絡む場合は **テスト一覧をデシジョンテーブルで設計** し、なぜその
組合せを選んだかを残す (= レビュー時の議論材料、追加観点が出た時の足場)。

## 適用範囲

- **適用**: 実装コードの振る舞いを変える変更 (機能追加 / bug fix / 仕様変更)
- **除外**:
  - rule / 設定 / 文書 / one-shot script (テストハーネスを持たない / 持つ意義が薄い)
  - リファクタ単体 (= 振る舞い不変、既存テストで保護されていれば足りる)
  - typo 修正 / フォーマット整形

## Why

- **回帰防止**: テストが意図を凍結する
- **仕様の実例化**: テストは「動く仕様書」、ドキュメントより信頼できる
- **設計フィードバック**: テストが書きにくい = 設計に問題がある合図
