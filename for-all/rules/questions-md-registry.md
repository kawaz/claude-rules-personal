# 裁定待ちは docs/QUESTIONS.md に常時集約

## AskUserQuestion 使用禁止

**Why**: タイムアウト無しで無限待ち、リモート応答不可、ユーザ入力中の割り込み不快、総じてUX悪い

## docs/QUESTIONS.md
ユーザ裁定が必要な確認事項 (設計分岐・命名・不可逆操作の可否など) は、チャット提示と**同一ターン**でプロジェクトの `docs/QUESTIONS.md` に記録する。
チャットで「👺TNG-Q1 の裁定お願いします」のように **👺 + ラベル ID** で参照し、質問の正本はファイルが持つ。
回答は「SEC-Q6=a」のようにラベル + 選択肢記号だけで通じる形にする。

## ファイル規約

- 場所: リポ管理プロジェクトの `docs/QUESTIONS.md` (commit する)。リポの無い作業では本運用は適用外
- 粒度は**索引**: ラベル + 質問 1〜2 行 + 選択肢と AI の推し (根拠 1 文) + 参照 (リポ相対パス + 節 / 行番号)。詳細の正本は findings / issue / DR 側に置き、複製しない
- ラベルはバッチ毎に一意な短プレフィクス (TNG-Q1, IMPL-Q3, SEC-Q1 形式。Qn だけの様な使い回し禁止)
- **裁定依頼は「👺ID の裁定お願いします」の形** 絵文字 👺 は固定。ユーザは正規表現 `👺\s*[A-Za-z0-9α-ωΑ-Ω\-]{2,}` で裁定依頼マーカーをハイライト及びアラーム設定しており、裁定依頼を目視でも機械的にも発見しやすい。
- **裁定依頼以外では 👺 を付けない** 裁定済みや過去の参照として 👺ID を使用すると誤陽性となり、ルールの目的に反する。
- ファイル冒頭に本運用規約を数行で自記する (ゼロコンテキスト読者・並行セッション向け)

## 更新規律

- Q の提示と同一ターンでファイル更新 + path 指定 commit (push は通常のリリース窓に同乗)
- 裁定が下りたら該当セクションを**削除**し、裁定内容は正規の記録先 (DR / issue / journal / close_reason) へ反映する。ファイルは常に「現在待ち」だけを持つ (経緯は git log と各記録先が担う)
- ユーザの回答が新しい Q を生んだ場合 (前提棄却・逆質問)、旧 Q の削除と新 Q の追記を同時に行う
- 調査中で提示に至らないものは Q として載せない (載せるのはユーザが今答えられる形になったものだけ)

## 説明要求への応答はファイル内セクション補充で

ユーザが Q に対し「説明して」「詳しく」「それ何?」と返したら、**チャットで長々説明しない** (TL に流れて参照コストが高い)。当該 Q を QUESTIONS.md 内で「### 背景説明」等のサブセクション付きで**書き直して再提示** (背景・機構・具体例を self-contained に) し、チャットは「👺HO-Q5 を説明付きで再提示しました」の参照だけ返す。機構の前提知識が要る Q は初回提示時から説明セクションを添えるのが望ましい (kawaz 確認「説明が TL に流れず良い感じ。とてもやりやすくなった」)。

## Why

- ユーザが「何待ちか」をチャット履歴から掘り返す手間削減
- AI にリストと説明を再生成させるコストの削減（今何待ち？を撲滅）
- セッション境界・並行セッションを越えて裁定キューが共有される
