# 統合論 (= ハイブリッドが正解) への警戒

複数案 / 複数スタイル / 複数手法を比較する場面で、AI (= 自分) は **「両方の長所を 1 つに集約する案 (= ハイブリッド)」が正解** と整理しがち。
この衝動は **両者の長所が排他的な制約から導かれている場合に集約不可能** という事実を見落とす self-blindspot。

## 症状 (= 書きたくなった瞬間)

- 「両方の良いとこ取り」「ハイブリッド」「統合案」「中間 / 折衷」
- 「A の長所 X と B の長所 Y を組み合わせれば最強」
- 比較表を作って「両方の ✓ を持つ第 3 案」を結論にする

## なぜ駄目か

- 各案の長所は **その案を選んだ制約** から導かれている (= domain 性質 / トレードオフの取り方 / 責務分担 / 適用範囲)
- 制約自体が **互いに排他的** な場合、長所を集約すると **両方の制約を侵害** して両方の長所が崩れる
- 結果として「両方の良いとこ取り」案は **両方の悪いとこ取り** になる

## 例外 (= ハイブリッドが正当な場面)

- 各案の長所が **互いに独立した次元** から出ている (= 直交している)
- 集約案で両元案の制約を **両方とも満たせる** ことが確認できる
- 「足し算で 100 %」が成り立つ証跡がある (= sample 実装で確認可能)

これらが成立する場合のみハイブリッド提案は OK。確認なしのハイブリッドは禁則。

### 責務境界明示分離パターン (= 両 style 共存が正当な具体パターン)

「どちらの style に属するか」が **明示分離** できる場合、両 style が同じ場所に共存しても集約問題は起きない。分離パターンは 2 つ:

1. **case 単位分離**: case ごとに「どちらの style に属するか」が明確で、case 単独で自分の style を取る (例: CLI ツールで「POSIX 準拠 case」と「ツール独自判断 case」が別 case として並ぶ。kawaz/die の `tests/run.sh`)
2. **同一 case 内文章内分離**: 同じ case の文章内で「外部仕様引用」と「独自判断 marker」が明示分離 (例: `# /dev/null is a char device but NOT a TTY (POSIX isatty(3) returns false; ...). Runs unconditionally on all OSes under DR-0008.` のように、文章構造で責務境界を区別)

「同一 case 内で複数 style を **無自覚に** 混ぜたい」と思った時は危険信号 (= 「ハイブリッド統合」誘惑)、「**明示分離を意図して書く**」のなら正当。判断軸は「読み手が style 切替点を視認できるか」。

## AI 自警 (= ハイブリッド案を書きたくなった瞬間に自問)

1. **各長所の由来は何か?** (= どの制約 / domain 性質 / トレードオフから出ているか)
2. **その由来は集約案でも保持されるか?** = 集約することで侵害される制約はないか
3. **集約案を採用した時、両元案の長所はそれぞれ何 % 維持されるか?** = 「足し算で 100 %」ではなく「両方 50 % 以下」になるなら集約は不利益

3 つすべてを答えられないなら、ハイブリッド案を書かない。各案を **そのまま並列** に提示してユーザに選んでもらう。

「両方の良いとこ取り」「ハイブリッド」「統合案」「折衷」と書く瞬間に手を止める。

## 関連事例

- **テストスタイルの統合論失敗** (= 2026-06-29 grapheme.mbt と timespec.mbt の相互レビュー時): grapheme は外部 source of truth (UAX #29) への準拠 domain、timespec は内部 source of truth (自分で決める) domain。両者の長所 (= 構造化テンプレ + 仕様 ref / ドメイン語彙固定 + 自然文段落) は **domain 性質が互いに排他的** な制約から導かれていて、ハイブリッドにすると両方の domain 性質を裏切る。kawaz の「あちらとこちらは扱ってるドメインが違う」指摘で明らかになった
- **責務境界明示分離パターンの発見** (= 同日 die session との 3 者比較): kawaz/die は CLI ツールで「POSIX 慣習準拠面」と「ツール独自判断面」の両者を負う domain だが、case ごと / 同一 case 内文章内のいずれかで責務境界を明示分離する慣習 (= rule 改訂前から先取り実装) で両 style 共存。「ハイブリッドは壊れる」警告は **library 系の同一 case 内で style 選択を強制される場合** に強く適用される話で、CLI 系のように責務境界が分離可能な場合の例外パターンが見つかった
