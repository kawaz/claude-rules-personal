# TDD とテスト設計

機能追加・bug fix では **t_wada の TDD** + **テスト設計の核心原則** を実践する。
テストコードは **真の仕様書** として書く。

## サイクル (RED / GREEN / REFACTOR)

1. **RED**: 仕様の輪郭を明確化する失敗テストを書く
2. **GREEN**: 最小実装でテストを通す
3. **REFACTOR**: テストが通った状態を保ったまま改善

## 核心原則 — テストが「何を保証したいか」を明確にし、仕様輪郭を網羅描画する

テストコードを書く / 読む / 評価する **すべての場面**で常に念頭に置く:

> **テストの最重要軸**: 「**何を保証するのか / したいのか**」「**テストで仕様や正常状態の輪郭を網羅的に表現できているか**」。テスト種類 (unit / e2e / fuzz / bench 等) の選定や追加判断より優先する。

「カバレッジ数値を上げる」「とりあえず代表 case」では不十分。test を「動く仕様書」として書くなら、その「仕様」がテスト全体で **正確かつ網羅的に描けているか** が品質の本質。

### How to apply — どんなテスト作業でも以下を自問

1. **このテストは何を保証する?** (= 1 文で言える)
   - 仕様準拠? 退行検知? 上流変化検知? 性能保証? セキュリティ?
   - 「**何**を**いつ**から**いつ**まで保証する」を明示
2. **正常状態の輪郭は描けているか?**
   - 期待される全パターン (= open ケース) を網羅
   - 期待される全境界 (= close ケース) を網羅
   - 「代表的」で済ませず、**意味論的に独立な軸を全列挙**
3. **異常系の輪郭は描けているか?**
   - 入力空間の境界 (空 / max / 不正型 / 越境)
   - 失敗 mode の独立な軸を全列挙
4. **そのテストはどこで行うべきか?** (= 種類選定は輪郭描画の **後**)
   - 純粋ロジック → unit (= 速い、隔離、大量)
   - 観察可能振る舞い → e2e (= 環境依存含む)
   - 退行検知 → regression test
   - 上流変化 → nightly / dependabot
   - 「e2e でしか測れないもの」を e2e に、「unit で済むもの」を unit に
5. **何を assert するか?** (= 厳密 vs loose)
   - 仕様が要求するものだけを assert、過剰は実装拘束に
   - 例: byte 厳密 (= 完全一致が仕様) vs 意味論 loose (= 「表示崩れない」程度が仕様、実装は OS / runtime ごとに違って OK)
6. **追加判断より優先**: 「何のテストを追加するか」の前に「**今あるテスト群が仕様輪郭を網羅できているか**」を見直す。隙があれば埋める、既存と重複なら追加しない

## RED 段階で必ず意識すべき観点

「ただ失敗するだけのテスト」を量産しない。**こうなるべき仕様 (ゴール) を表現する** こと。
惰性で穴だらけのテストを並べるのは最悪 (= 通っても仕様が担保されない、後で書き直し)。

最低限すべて意識し、該当しない観点は「該当なし」と明示判断:

- **正常系**: 主要ユースケースが期待通り動く (= ゴールの達成)
- **異常系**: 不正入力 / 失敗パスで適切に失敗する (= 黙って成功で済ませない、エラー型・メッセージも仕様)
- **境界ケース**: off-by-one / 最小値・最大値 / 空入力 / null・undefined / 長さ上限・桁上限 / 0・負数・小数
- **エッジケース**: タイムゾーン / ロケール / unicode (surrogate pair・絵文字・結合文字) / 浮動小数 precision / 桁あふれ / 環境差分
- **複数条件 (デシジョンテーブル)**: 条件の組合せを表で網羅。independent な n 条件で 2^n の組合せが出るので必要組合せを精選 (ペア網羅 / 危険なペア優先)。「全部 OR / 全部 AND」だけで済ます手抜きを避ける
- **同値分割**: 入力空間を「同じ振る舞いになる集合」に分け、各集合から代表値を 1 つずつ。境界ケースと組み合わせて入力空間を覆う
- **状態遷移**: ステートマシン的振る舞いがあれば、各遷移 / 不正遷移 / invariant を確認
- **副作用 / べき等性**: 同じ操作を繰返したとき同じ結果か、副作用の有無、ロールバックの可能性
- **並行性 / レース**: 並列実行で正しく動くか (共有状態 / 競合 / デッドロック / 順序保証)
- **回帰**: 過去の bug 報告 / 修正済みケースが再発しないか
- **性能境界** (該当時): 大入力でタイムアウト / メモリ越えしないか
- **セキュリティ境界** (該当時): injection / 認可越境 / 入力エスケープ / 権限昇格パス

複数条件が絡む場合は **テスト一覧をデシジョンテーブルで設計** し、なぜその組合せを選んだかを残す
(= レビュー時の議論材料、追加観点が出た時の足場)。

## テストの全体観 (= チェックリスト)

| 観点 | 例 |
|---|---|
| 純粋ロジック | trim / join / parse |
| 観察可能振る舞い | CLI argv 解析、stdin/stderr 経路、exit code |
| 外部統合 | homebrew / dependency tree / API contract |
| 非機能 | size / cold start / throughput threshold |
| 環境互換 | OS / language runtime version matrix |
| 時間軸退行 | nightly / dependabot / scheduled e2e |
| セキュリティ | dependency audit, fuzz, SAST |
| 本番監視 | synthetic monitoring (= cron で本番に対し定期 e2e) |

各観点で:
- **保証したいことは?** (= 1 文)
- **輪郭網羅できてる?** (= 意味論的な軸を全列挙)
- **どこで実行?** (= local / pre-commit / CI on PR / CI nightly / release CI / 本番監視)
- **いつ実行?** (= save 毎 / commit 毎 / push 毎 / 定期 / on-incident)
- **何を assert?** (= 厳密 vs loose、過剰拘束を避ける)

## テストコード = 真の仕様書 (= コメント inline 化)

> 各テストケースは何をテストしているのか意図や判断をテストケースのコメントに込める。DR は議論ログとして今もあるが、それをテストケースにも適用しておくと、テストが仕様であると言う形に出来る。理想的には完璧なテストセットがあれば言語移植も簡単だし、リファクタやサイズ最適やパフォーマンス最適も安心して出来るし性能向上にもつながる。完璧に同じ振る舞いが定義できていればソースコードがなくてもゼロから再現ができるのが理想。

**核**: テストコード = 真の仕様書。**「なぜこの入力か」「なぜこの期待結果か」「どの DR の判断に基づくか」をテストケース直近のコメントに込める**。

これにより:
- ソースコードを捨てても test だけで実装をゼロから再現可能
- 別言語移植も test を pass させれば済む
- リファクタ / 最適化 / 性能改善も test 全 pass で safety 確保
- 後続セッション / 別エンジニアが test を読むだけで仕様 + 判断経緯を理解可能 (= DR を別タブで開かなくて済む)

### コメント様式 (= 推奨パターン)

- **全 case に意図コメントを書く (= 「自明だから省く」は禁則)**: プロジェクトに浸かった立場の自明はゼロコンテキスト読者 (= 別言語移植者 / OSS 公開での初見 / 半年後の自分) には自明でない。例えば `parse_duration("5m") -> Duration(300000)` は ms 換算規約 + `m=分` の単位規約が暗黙に効いていて、明文化されないと再現できない。コメント省略の禁則であって、test 自体の取捨 (= 下の「対極」section) とは別の話。迷ったら書く (= 過剰コメントより省略の方がリスクが高い)。
- ケース直前に意図を書く: 「**何の入力で何を確認、なぜそうなるべきか (= DR / 仕様根拠)**」。長さは説明に必要なだけ — 単純な case は 1 行、複雑な条件 / 境界判断 / 仕様根拠の絡む case は段落で書いて構わない (= 行数を絞るために必要な説明を削るのは本末転倒)。
- 複数 case をまとめた section の冒頭に共通の判断を書く (= 軸の意味論)
- **DR の判断本質を文章で書き、DR ref は補助 marker**: 「`// 末尾 @ も先頭 @ と同等に Absolute フラグ立てる: @ idempotent 規約による (DR-0007 Phase 2 marker)`」のように、test 読者が **DR を別タブで開かなくても判断が完全に分かる** 形にする。`// DR-0006: -n is cat-equivalent` のような DR ref 単独引用は不可 (= DR が消えた / 別プロジェクトに移植 / OSS 公開 した時に意味を失う)。**portability の根拠**: 「テスト = 真の仕様書、ソースを捨てても再現可能、別言語移植も test を pass させれば済む」を満たすため、test 内の reference は test 単体で self-contained でなくてはいけない。DR ref は「この判断は記録された経緯がある」という意図的な history pointer 扱いで、必須ではなく補助情報。同じ原則は issue ref / PR ref / commit hash 等の全 cross-reference に適用
- 境界判断は **なぜそこが境界か** を明示 (例: `// 6 ASCII whitespace bytes only — Unicode WS (NBSP, U+2028) intentionally not trimmed per DR-0001`)
- 仕様変更で test を変える時、**変更の議論ログも近接 comment に追記** (例: `// DR-0005 で --eol 廃止に伴い -n の意味を cat 同等に refine`)

## 適用範囲

- **適用**: 実装コードの振る舞いを変える変更 (機能追加 / bug fix / 仕様変更)
- **除外**:
  - rule / 設定 / 文書 / one-shot script (テストハーネスを持たない / 持つ意義が薄い)
  - リファクタ単体 (= 振る舞い不変、既存テストで保護されていれば足りる)
  - typo 修正 / フォーマット整形

## 対極 — テストを書かない / 削る / 緩める判断 (= self-blindspot)

「テストを書け」「網羅しろ」だけだと過剰になる。対極の判断も持つ:

- **重複なら追加しない**: 既存 test が同じ仕様輪郭をカバー済なら追加しない (= テスト群肥大化の防止)
- **意味のある軸でないなら追加しない**: 「`-n` × `--sep` の組合せ」が直交 (= 互いに影響しない) なら、組合せ test の網羅価値は低い。**直交性自体を仕様として 1 ~ 2 case で固定** すれば十分
- **過剰 assert は緩める**: 仕様が「表示崩れない」なら byte 厳密一致を assert しない (= 実装拘束、OS 跨ぎで誤検知)
- **「該当なし」を明示する勇気**: 全観点を埋める必要はない、該当しない観点は「該当なし: 理由」とコメントで残す (= 後続が「漏れ」と誤解しないため)。[[no-historical-noise]] の「除外リスト禁止」とは文書種別で棲み分ける (= 優先順位の正本は [[rule-writing-guidelines]])
- **古い test の削除も品質**: 仕様変更で意味を失った test、または `[[test-failure-no-tampering]]` で `#[ignore]` 化されたまま追跡 issue が closed なら **削除する**

## やりがちな失敗 (= 反例)

- 「カバレッジ % を上げる」を目的に意味論抜きの test 追加
- e2e で全パターン展開 (= unit で済むものを e2e でやる、環境依存が乗る)
- 「代表 1 case」で 終わらせ境界 / 異常系を放置
- byte 厳密 assert を仕様要件以上に書いて実装拘束 (= 例: default mode で OS 跨ぎ byte 一致を仕様化してしまう)
- テスト追加を「対応終わった」の指標にする (= 仕様輪郭描画の完全性で判断するべき)

## Why

- **回帰防止**: テストが意図を凍結する
- **仕様の実例化**: テストは「動く仕様書」、ドキュメントより信頼できる
- **設計フィードバック**: テストが書きにくい = 設計に問題がある合図
- **言語 / 実装中立**: テストが完璧なら言語移植・リファクタ・最適化が safety net 上で可能
- **後続セッションの省コスト**: コメント inline により DR を別タブで開く必要なし、test 読むだけで仕様 + 判断経緯が理解可能
