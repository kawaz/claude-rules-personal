# タスクオーケストレーション (Phase 0-4)

行動制約 (禁則) は常時ロードの rules が担う。本 skill が担うのは
**実行順序の制御**: 「次に何をやるか」を惰性でなく判断で選ばせる。

## 三原則 (全 Phase の判断基準)

1. **完了条件を先に固定する** — 検証方法が書けないタスクは、まだ理解できていない
2. **主張ではなく観測を信じる** — 「動くはず」は証拠ではない
   (empirical-verification と同根、本 skill は適用順序を担当)
3. **次の一手は「いま一番リスクが高い場所」で選ぶ** — 元の計画の次の行ではなく

## 適用ゲート

以下は本 skill の対象外 (= Phase を踏まず直接やる):

- 軽微編集 (work-principles の「メインが直接行ってよい作業」に該当するもの)
- 議論レーンの調査 / PoC (discussion-style — 議論の一部としての小実験は自由)
- 手順が確定済みの runbook 実行

迷ったら適用する (= 過剰プロセスのコスト < 完了条件なし着手のコスト)。

## Phase 0: アンカー3行 (何かに触る前)

TODO 作成 (work-principles の TODOファースト) の**前**に、以下3行を書く:

```
完了条件: <観測可能な形で。「〜が動く」でなく「コマンド X が出力 Y を返す」>
検証方法: <コマンド / diff / 実行結果。何を見れば完了と判定できるか>
やらないこと: <今回スコープ外にする隣接事項>
```

- **検証方法の行が書けない = 理解不足**。書けるようになるまで先に調査する
  (このゲートが Phase 0 の存在意義。3行を埋める作文作業ではない)
- スコープが膨らみかけたら、この3行と照合する (= 錨)
- 3行はユーザへの最初の応答 or TODO 先頭に置き、セッション中参照可能にする

## Phase 1: 偵察と3値仕分け

計画の前に実物を軽く読み、読んだ情報を3値でラベリングする:

- **事実**: 自分が観測した (コマンド出力 / ファイル内容 / 実行結果)
- **仮定**: 誰かの主張・推測 (ユーザの見立て、コードコメント、issue 本文、
  過去ログ、自分の記憶)。もっともらしくても観測するまで仮定
- **不明**: どちらでもない。埋める必要があるかをリスク順で判断

**一番危険なのは事実のふりをした仮定**。特に「〜が怪しい」という誘導
(ユーザ発、過去セッション発を問わず) は仮定としてマークし、検証対象に
するだけで、探索範囲をそれに限定しない。

## Phase 2: リスク順分解

概念でなく**独立して検証できる単位**で切り、順序は**リスク順**:

- 最初にやるのは「外れたら計画全体が崩れる仮定の検証」
- **簡単なところから片付けて勢いをつけるのは罠** — 核心が不成立なら
  簡単な部分の成果ごと捨てることになる
- 各単位に「この単位の検証方法」を付けてから着手 (= Phase 0 のフラクタル)
- サブエージェント委譲の判断はここで行う (work-principles の tier 別基準)

## Phase 3: 実行ループ

1ピースずつ進め、**その場で検証**する (最後にまとめて検証しない —
失敗の発見が遅れるほど原因候補が増える)。

各ピース完了ごとに4つの自問で次の一手を選び直す:

1. **観測は計画と矛盾していないか** — 矛盾したら計画側を疑う (計画に観測を
   合わせるのは test-failure-no-tampering と同型の隠蔽)
2. **いま最大のリスクはどこか** — Phase 2 の順序は暫定。観測で更新する
3. **次の行動は可逆か** — 不可逆操作 (削除 / push / 破壊的変更) は根拠の
   密度を一段上げる (retreat-is-last-resort)
4. **人間にしか答えられないことで詰まっていないか** — 詰まっているなら
   即問う。推測で代行しない

行動 vs 追加調査の切替は **手戻りコストと確認コストの比較**で決める:
確認が1コマンドで済むなら確認、手戻りが軽微 (= 可逆 + 局所) なら行動。

## Phase 4: 全体検証と報告

1. **別の層で確認する** — コードなら実行、文章なら初見読者として読む、
   設定なら再現環境で適用。作った層と同じ層の再読は検証ではない
   (kawaz/die の教訓: unit test が本番経路 `buildArgOutput` を通らず green)
2. **壊しに行く** — 空入力・境界値を実際に当てる (境界観点の網羅は
   tdd-and-test-design 側が正)
3. **原因の寄与証拠** — 報告に書く各原因に「症状へ寄与した証拠」を1つずつ
   持つ。なければ**「寄与は未確認」と明記**する。コードが直っていても
   説明が観測と矛盾していれば、その報告は嘘になる
4. **依頼原文との突き合わせ** — Phase 0 の3行と最終成果を照合。
   「やらないこと」に入れた事項の扱いも報告に含める
5. **diff 通読** — 意図しない変更の混入がないか

## 常時ロード rules との分担 (重複させない)

本 skill は順序制御のみ。以下は rules 側が正 (本 skill 内の言及は
適用タイミングの指示であって内容の再定義ではない):

- 観測の方法論・マトリクス検証・観測道具 → empirical-verification
- テスト設計の網羅観点 → tdd-and-test-design
- test 改変禁則 / flaky の説明責任 → test-failure-no-tampering
- 撤退の順序 → retreat-is-last-resort
- ハイブリッド誘惑 / デフォルト収束 → synthesis-temptation-guard /
  default-convergence-guard
