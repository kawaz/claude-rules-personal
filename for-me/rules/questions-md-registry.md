# 裁定待ちは docs/QUESTIONS.md に常時集約

ユーザ裁定が必要な確認事項 (設計分岐・命名・不可逆操作の可否など) は、チャット提示と**同一ターン**でプロジェクトの `docs/QUESTIONS.md` に記録する。チャットは「VF-Q 待ちです」のようにラベル参照だけで済ませ、質問の正本はファイルが持つ。回答は「SPL-Q6=a」のようにラベル + 選択肢記号だけで通じる形にする。

## ファイル規約

- 場所: リポ管理プロジェクトの `docs/QUESTIONS.md` (commit する)。リポの無い作業では本運用は適用外
- 粒度は**索引**: ラベル + 質問 1〜2 行 + 選択肢と AI の推し (根拠 1 文) + 参照 (リポ相対パス + 節)。詳細の正本は findings / issue / DR 側に置き、複製しない
- ラベルはバッチ毎に一意なプレフィクス (VF-Q1, SPL-Q3, CR-Q1 の形式。Qn の使い回し禁止)
- ファイル冒頭に運用規約を数行で自記する (ゼロコンテキスト読者・並行セッション向け)

## 更新規律

- Q の提示と同一ターンでファイル更新 + path 指定 commit (push は通常のリリース窓に同乗)
- 裁定が下りたら該当セクションを**削除**し、裁定内容は正規の記録先 (DR / issue / journal / close_reason) へ反映する。ファイルは常に「現在待ち」だけを持つ (経緯は git log と各記録先が担う)
- ユーザの回答が新しい Q を生んだ場合 (前提棄却・逆質問)、旧 Q の削除と新 Q の追記を同時に行う
- 調査中で提示に至らないものは Q として載せない (載せるのはユーザが今答えられる形になったものだけ)

## Why

- ユーザが「何待ちか」をチャット履歴から掘り返す / AI にリストと説明を再生成させるコストの削減 (kawaz 提案 2026-07-13、kuu で実証済み — 「とてもやりやすくなった」2026-07-14)
- セッション境界・並行セッションを越えて裁定キューが共有される (ファイルがリポにあるため)
