# Dogfooding での気づきは上流の kawaz 製ツールへ還元する

kawaz 製ツール (bump-semver, ccmsg, hyoui, cache-warden...) を別プロジェクトで
使っていて仕様の罠・bug・改善点に気づいたら、**利用側プロジェクトの docs (journal/findings)
にとどめない**。kawaz は自製ツールをいくらでも直せるので、利用側に埋もれた気づきは
改善機会の損失になる。

## 還元先 (どちらか、両方も可)

1. **ツールリポの `docs/issue/` に起票** (kawaz リポは GH Issues でなくローカル issue 運用)。
   ツールリポ側の記法・命名・status 語彙に合わせる。
2. **当該ツールを担当中の Claude セッションに cmux-msg で直接共有** (セッション ID が
   分かっている / kawaz から指示がある場合)。即時対応が見込めるならこちらが速い。

## 起票・共有に含める内容

- 現象 / 再現コマンド (コピペで再現できる粒度)
- 実機確認の結果 (推測でなく観測)
- 利用側で取ったワークアラウンド
- 改善提案 (任意。「エラーメッセージにヒント追加で十分」等の所感も価値がある)

## 部外者として起票するときの姿勢 (= フラグ止まり、実装判断は当事者に委ねる)

利用側 = **部外者セッション**は対象ツールの責務・コンテキストを把握していない。部外者が具体的な
実装指示を出すと、当事者がそれを鵜呑みにして的外れな実装に進むリスクがある。だから起票は:

- **「フラグ + 一次資料の提示」に留める**: 「この機能/手法が使えそう」「賢い所と、そうでなくなる
  条件はここ (= 一次資料 URL/path) を見て確認して」まで。**実際の実装にどう適用するかは当事者
  セッションに考えさせる**
- **断定しない・鵜呑み防止**: 部外者の観測も推測混じりかもしれない前提で「裏取りしてから採否を
  決めて」と明記する。具体的な diff やコードを書いて渡さない
- **該当性を自分で確認してから起票**: 的外れな issue は当事者の時間を奪うノイズ。起票前に対象の
  コードを読み、本当に該当するか確認する (例: コマンド名マッチ機能を、クォート解析が責務の hook に
  提案しない)

## How to apply

- 気づいた時点で還元する (まとめて後で、にしない — 忘れる)
- 利用側の journal には「上流に起票済み: <repo> docs/issue/<file>」の相互参照を残す
- ツールが kawaz 製かは repo owner (github.com/kawaz/*) で判定。迷ったら kawaz に確認
