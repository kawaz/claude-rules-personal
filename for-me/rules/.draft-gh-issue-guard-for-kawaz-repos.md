## 背景

kawaz の運用ルール:
- `kawaz/*` 自己所有リポでは **未解決 issue を `docs/issue/<file>.md` でローカル起票**
- GitHub の Issues 機能は使わない (= ローカル file 運用)
- 参考: `docs-knowledge-flow.md` ルール

しかし Claude (agent) は default で `gh issue create` を使いがち = 違反しやすい。
実例: bump-semver セッションで Claude が `gh issue create --repo kawaz/claude-gh-monitor` を実行
(= 当時 docs/issue/ 無し → kawaz が「ここは gh issue でいい」と許容、しかし 2026-06-02 時点で
claude-gh-monitor は docs/issue/ 採用済 = 当初の例外論拠は失効)。

## 方針: 所有者ベースのシンプル判定

「allow list を保守する」モデルは捨てる。**所有者だけで分岐する**:

| owner | 判定 | 振る舞い |
|---|---|---|
| kawaz (個人 OSS 本体) | guard 対象 | block + `docs/issue/<slug>.md` redirect hint |
| kawaz 個人面 overlay 系 (= `kawaz/claude-rules-*` 等) | guard 対象 | 同上 (= 全部 owner=kawaz なので 1 列で覆われる) |
| kawaz123 (= emeradaco 業務面) | guard 対象外 | 通常通り (= 業務 issue は GH Issues) |
| その他第三者 (= 外部 OSS、非 kawaz owner) | guard 対象外 | 通常通り |

「kawaz が自己所有 (= kawaz owner)」ならローカル起票、それ以外なら通常通り、というだけ。
docs/issue/ ディレクトリの実体有無は判定に使わない (= 「未採用だが運用予定」のリポでも guard
を効かせて誘導したほうがルール定着に資する)。

### kawaz 自己所有 owner の列挙 (= 判定 list)

現時点で「kawaz owner」と判定すべきもの:

- `kawaz` (= 個人 OSS 本体 + 個人面 overlay リポ群すべて)

list を将来増やすか確認: kawaz 個人面で持つ overlay 系リポ (`kawaz/claude-rules-personal`,
`kawaz/claude-rules-syun`, `kawaz/claude-rules-zunsystem` 等) はすべて owner = `kawaz` なので、
owner 1 種類で全部カバーできる (= 余計な list 保守不要)。

将来別の overlay account (= 仮に `kawaz-owner-X`) を持ったら、その owner も判定 list に追加する。

### 業務面 (kawaz123/*) の扱い

- owner = `kawaz123` は **guard 対象外** = `gh issue create` 通常通り
- 業務リポ (emeradaco) の運用は GH Issues 主体、kawaz123 個人作業も慣習的に GH Issues 利用
- 同 hook で `owner != kawaz` なら早期 exit、別 hook を作らない (= シンプル)

## 採用ポリシー: 候補 A + C (= 確定)

> Decision: 候補 A (= hard block) + 候補 C (= env flag override 専用) の組合せで進める (kawaz 承認済)。
> 候補 B (= warn + 確認プロンプト) は採用しない。

### 候補 A: hard block

| 観点 | 評価 |
|---|---|
| 安全性 | ◎ 違反は完全に防げる |
| 体感 | △ block 通知が出る → Claude は別 path (= docs/issue/) を試す = redirect 効きやすい |
| 突破口 | env flag `CLAUDE_ALLOW_GH_ISSUE=1` を 1 回ずつ明示で許可 |

### 候補 C: env flag override 専用 (= allow-list 完全廃止)

| 観点 | 評価 |
|---|---|
| 安全性 | ◎ 明示 opt-in が必要 |
| 体感 | ◎ list 保守不要 |
| 突破口 | `CLAUDE_ALLOW_GH_ISSUE=1 gh issue create ...` のみ通過 |

### 候補 B (= 不採用): warn + 確認プロンプト

| 観点 | 評価 |
|---|---|
| 安全性 | △ 確認をスルーすると違反成立 |
| 体感 | × AskUserQuestion フローが増える、テンポ悪化 |
| 突破口 | 確認 yes/no、運用上は block と大差なし |

### 採用根拠

- 候補 A 単体だと「kawaz/claude-gh-monitor の併用フェーズ」のような過渡期で詰む
- env flag override (= 候補 C) を付与しておけば、過渡期も「flag 付きで明示」で進める
- 静的 allow-list は持たない (= owner 1 種で判定、保守ポイント 0)

deploy 経路:
- `~/.claude-personal/settings.json` の `extraKnownMarketplaces.push-guard` で `github:kawaz/claude-push-guard` を登録
- 同 `enabledPlugins["push-guard@push-guard"]: true` で enable

### 結論: 本 guard も独立 plugin リポ化する

理由:
- push-guard、force-bun、force-uv、claude-rules-source-guard、bash-safety 等の既存 PreToolUse hook は **全部 plugin リポ化** されている (= 一貫パターン)
- rule リポ (`claude-rules-personal`) に直配置すると hooks の自動デプロイ口がなく、settings.json を手書きしないと load されない (= 自動更新も効かない)
- plugin リポ化すれば marketplace 経由で auto-update も効く

→ **新規リポ `kawaz/claude-gh-issue-guard` を作成** し、push-guard と同じ structure で実装する。

### `hooks/gh-issue-guard.sh` (sketch)

```bash
#!/bin/bash
# PreToolUse(Bash) hook: block `gh issue create` (and aliases) toward kawaz-owned repos.
# Routes Claude to docs/issue/<slug>.md local tracking (= docs-knowledge-flow.md rule).
#
# stdin: JSON { tool_input: { command: "<bash command>" } }
# exit 0 = proceed, exit 2 = block
#
# NOTE: push-guard と同じ流儀で `set -e` は使わない。hook 自体の不具合は通過 (= ユーザの作業を巻き込まない)。

# Env flag override (= ad-hoc opt-in, 過渡期 / 第三者連動起票用)
[ "${CLAUDE_ALLOW_GH_ISSUE:-0}" = "1" ] && exit 0

input=$(cat)
cmd=$(printf '%s' "$input" | jq -r '.tool_input.command // empty' 2>/dev/null) || exit 0
[ -n "$cmd" ] || exit 0

# Match `gh issue create` and LLM-likely aliases.
# 採用: (create|new|open)
#   - create = canonical (gh の正規サブコマンド)
#   - new    = よくある LLM 揺れ (= `gh issue new` を打ちがち、実在しない)
#   - open   = 念のため (= GitHub UI 的に「Open issue」と書かれているため LLM が引っ張られる)
# 不採用: add / make / file (= 観測例なし、false positive リスクの方が大)
# 行頭 / `&&` / `;` / `||` / `|` 直後のみマッチ (= heredoc 内文字列を誤検知しない)。
if ! printf '%s' "$cmd" | grep -qE '(^|&&|;|\|\|?)\s*(env\s+[A-Z_]+=[^ ]+\s+)*gh\s+(.*\s+)?issue\s+(create|new|open)\b'; then
  exit 0
fi

# Determine target repo:
#   1. --repo <owner>/<repo> 引数優先
#   2. なければ cwd の git remote origin URL を解析
repo=$(printf '%s' "$cmd" | grep -oE -- '--repo[= ][^ ]+' | head -1 | sed -E 's/--repo[= ]//')
if [ -z "$repo" ]; then
  remote=$(git -C "${CLAUDE_PROJECT_DIR:-$PWD}" remote get-url origin 2>/dev/null || true)
  repo=$(printf '%s' "$remote" | sed -nE 's#.*github\.com[:/]([^/]+/[^/.]+)(\.git)?#\1#p')
fi
[ -z "$repo" ] && exit 0  # 判定不能 → 通す (= 過剰 block 回避)

owner=${repo%%/*}
name=${repo##*/}

# === 所有者ベース判定 ===
# kawaz 自己所有 owner 一覧 (= 個人 OSS 本体 + 個人面 overlay。全部 owner=kawaz)
case "$owner" in
  kawaz) ;;  # guard 対象、続行
  *) exit 0 ;;  # 第三者 / kawaz123 → 通す
esac

HINT_FILE="$(cd "$(dirname "$0")" && pwd)/gh-issue-redirect-hint.md"

cat >&2 <<EOF
BLOCK: \`gh issue create\` toward $owner/$name は直接実行できません。

kawaz 自己所有リポでは GitHub Issues を使わず、リポ内 \`docs/issue/<slug>.md\` で
ローカル起票する運用です (= docs-knowledge-flow.md ルール)。

- 起票: \`docs/issue/\$(date +%Y-%m-%d)-<slug>.md\` を作成
- 解決時のフロー / DR / runbook との使い分けは下記参照:

  ${HINT_FILE}

どうしても GH Issue が必要な過渡期 (= claude-gh-monitor のように移行中、または
第三者起票と連動する場合) は、明示的に env flag を付けて再実行してください:

  CLAUDE_ALLOW_GH_ISSUE=1 gh issue create ...
EOF
exit 2
```

### regex に含めた alias の根拠

| alias | 含める? | 根拠 |
|---|---|---|
| `gh issue create` | ◯ | canonical (gh の正規サブコマンド) |
| `gh issue new` | ◯ | LLM が打ちがち (= 他多くの CLI で `new` を使う、強い類推) |
| `gh issue open` | ◯ | GitHub UI で「Open issue」表記、LLM が引っ張られる可能性 |
| `gh issue add` | × | 観測例なし、false positive 寄り |
| `gh issue make` | × | 同上 |
| `gh issue file` | × | 同上 |

不採用の alias は false positive リスクの方が大きいので、運用してから観測例が出たら追加する方針。

### sketch 限界 (= 実装時に詰める)

- `gh api repos/.../issues` POST (= REST 直叩き) は未対応。Claude が REST 直叩きする頻度は
  低い、必要なら別 hook を追加

- 業務面 (CLAUDE_CONFIG_DIR=~/.claude-emeradaco) でも plugin が enable されていれば hook が走るが、
  owner=kawaz の case は業務面では稀 (= 越境作業時のみ)、走っても正しく動く

## Decision points (= 確定/未確定)

### 確定 (= 2026-06-02 kawaz 承認)

1. **guard 強度**: 候補 **A + C** (= hard block + env flag 突破) で進める
2. **guard 対象 owner**: `kawaz` のみで足りる (= overlay 系も全部 owner=kawaz)

   - rule リポに直置きすると settings.json 手書き必要 + auto-update も効かない
6. **regex の alias 範囲**: `create | new | open` の 3 種を含める (= LLM の揺れ吸収、`add / make / file` は不採用)
