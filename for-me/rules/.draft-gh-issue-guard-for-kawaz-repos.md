## 方針: 所有者ベースのシンプル判定

「allow list を保守する」モデルは捨てる。**所有者だけで分岐する**:

| owner | 判定 | 振る舞い |
|---|---|---|
| kawaz (個人 OSS 本体) | guard 対象 | block + `docs/issue/<slug>.md` redirect hint |
| kawaz 個人面 overlay 系 (= `kawaz/claude-rules-*` 等) | guard 対象 | 同上 |
| kawaz123 (= emeradaco 業務面) | guard 対象外 | 通常通り (= 業務 issue は GH Issues) |
| その他第三者 (= 外部 OSS、非 kawaz owner) | guard 対象外 | 通常通り |

「kawaz が自己所有 (= kawaz owner)」ならローカル起票、それ以外なら通常通り、というだけ。
docs/issue/ ディレクトリの実体有無は判定に使わない (= 「未採用だが運用予定」のリポでも guard
を効かせて誘導したほうがルール定着に資する)。

## 仕様候補 (= 適用ポリシーの trade-off)

allow-list 軸ではなく、**guard の強さ** だけが軸:

### 候補 A: hard block (推奨)

| 観点 | 評価 |
|---|---|
| 安全性 | ◎ 違反は完全に防げる |
| 体感 | △ block 通知が出る → Claude は別 path (= docs/issue/) を試す = redirect 効きやすい |
| 突破口 | env flag `CLAUDE_ALLOW_GH_ISSUE=1` を 1 回ずつ明示で許可 |

### 候補 B: warn + 確認プロンプト

| 観点 | 評価 |
|---|---|
| 安全性 | △ 確認をスルーすると違反成立 |
| 体感 | × AskUserQuestion フローが増える、テンポ悪化 |
| 突破口 | 確認 yes/no、運用上は block と大差なし |

### 候補 C: env flag override 専用 (= allow-list 完全廃止)

| 観点 | 評価 |
|---|---|
| 安全性 | ◎ 明示 opt-in が必要 |
| 体感 | ◎ list 保守不要 |
| 突破口 | `CLAUDE_ALLOW_GH_ISSUE=1 gh issue create ...` のみ通過 |

### 推奨: **候補 A + C** (= block + env flag 突破)

- 候補 A 単体だと「kawaz/claude-gh-monitor の併用フェーズ」のような過渡期で詰む
- env flag override (= 候補 C) を付与しておけば、過渡期も「flag 付きで明示」で進める
- 静的 allow-list は持たない (= owner 1 種で判定、保守ポイント 0)

## hook 実装 sketch

### settings.json (PreToolUse for Bash)

```jsonc
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "hooks": [
          {
            "type": "command",
            "command": "$CLAUDE_CONFIG_DIR/hooks/gh-issue-kawaz-guard.sh"
          }
        ]
      }
    ]
  }
}
```

### `hooks/gh-issue-kawaz-guard.sh` (sketch)

```bash
#!/usr/bin/env bash
# PreToolUse hook: block 'gh issue create' toward kawaz-owned repos.
# stdin: JSON { tool_input: { command: "<bash command>" } }
# exit 0 = proceed, exit 2 = block

set -euo pipefail

# Env flag override (= ad-hoc opt-in)
[[ "${CLAUDE_ALLOW_GH_ISSUE:-0}" == "1" ]] && exit 0

input=$(cat)
cmd=$(printf '%s' "$input" | jq -r '.tool_input.command // ""')

# Match 'gh issue create' (= subcommand chain / env prefix 許容)
echo "$cmd" | grep -qE '(^|[;&|]\s*)gh\s+(.*\s+)?issue\s+(create|new)\b' || exit 0

# Determine target repo:
#   1. --repo <owner>/<repo> 引数優先
#   2. なければ cwd の git remote origin URL を解析
repo=$(echo "$cmd" | grep -oE -- '--repo[= ][^ ]+' | head -1 | sed -E 's/--repo[= ]//')
if [[ -z "$repo" ]]; then
  remote=$(git -C "${CLAUDE_CWD:-$PWD}" remote get-url origin 2>/dev/null || true)
  repo=$(echo "$remote" | sed -nE 's#.*github\.com[:/]([^/]+/[^/.]+)(\.git)?#\1#p')
fi
[[ -z "$repo" ]] && exit 0  # 判定不能 → 通す (= 過剰 block 回避)

owner=${repo%%/*}
name=${repo##*/}

# === 所有者ベース判定 ===
# kawaz 自己所有 owner 一覧 (= 個人 OSS 本体 + 個人面 overlay。全部 owner=kawaz)
guarded_owners=(kawaz)
# 将来 overlay account を別 owner で持ったらここに追加

is_guarded=0
for o in "${guarded_owners[@]}"; do
  [[ "$owner" == "$o" ]] && is_guarded=1 && break
done
[[ "$is_guarded" == "0" ]] && exit 0  # 第三者 or kawaz123 → 通す

# Block + redirect hint
cat >&2 <<EOF
[gh-issue-kawaz-guard] block: \`gh issue create\` toward $owner/$name.
kawaz 自己所有リポでは docs/issue/<slug>.md でローカル起票するルール (docs-knowledge-flow.md)。
- 起票: docs/issue/\$(date +%Y-%m-%d)-<slug>.md を作成
- どうしても GH Issue が必要 (= 過渡期、第三者起票の連動等): CLAUDE_ALLOW_GH_ISSUE=1 を付けて再実行
EOF
exit 2
```

### sketch 限界 (= 実装時に詰める)

- `gh issue new` (alias) 対応: 上記 regex に含めた (`create|new`)、要動作確認
- `gh api repos/.../issues` POST (= REST 直叩き) は未対応。Claude が REST 直叩きする頻度は
  低い、必要なら別 hook を追加
- regex の false positive (= heredoc / コメント内文字列で誤発火) → env flag で逃げられる
- `--repo $REPO` 等の変数展開 → 通す側にフォールバック (= 過剰 block 回避)
- 業務面 (CLAUDE_CONFIG_DIR=~/.claude-emeradaco) でも hook が走るが、owner=kawaz の case は
  業務面では稀 (= 越境作業時のみ)、走っても正しく動く

## 既存 rule との関係

- `docs-knowledge-flow.md`: `docs/issue/<file>.md` の **解決時フロー**を規定。本 hook は **入口を強制**
- `push-workflow.md`: push 時の品質 gate と同じ「違反は仕組みで止める」思想
- `release-flow-awareness.md`: tag/release を Claude が打たない、と同じ「自動化が打つ」思想
- `account-isolation.md` (overlay 経由、emeradaco): owner 判定で `kawaz` 自己所有のみ block、`kawaz123` (業務) は通す
- `claude-config-dir-isolation.md`: hook は `$CLAUDE_CONFIG_DIR/hooks/` 配下に置く想定 (= 環境ごとに override 可)
- `git-repo-management.md`: `kawaz` (public OSS) と `kawaz123` (業務 private) の所有者区別と整合

## Decision points (= kawaz 確認待ち、確認後の進路)

1. **guard 強度**: 候補 **A+C (= hard block + env flag 突破、推奨)** で良いか / B (warn) にするか
2. **guard 対象 owner**: `kawaz` のみで十分か、将来 overlay account を増やす予定があるか
   - 現状 overlay 系 (claude-rules-personal/syun/zunsystem) は全部 owner=kawaz、kawaz だけで足りる
3. **kawaz123 の扱い確認**: guard 対象外で良いか (= 業務面 GH Issues 利用は慣習通り)
4. **claude-gh-monitor 等の過渡期リポ**: env flag 突破運用で良いか、別途特例扱いするか
   - 推奨: 個別特例を設けず、env flag で都度突破 (= list 保守ゼロ維持)
5. **hook 設置場所**: `$CLAUDE_CONFIG_DIR/hooks/` 配下 (= 環境別) か、`~/.claude-personal/hooks/` 固定か
6. **`gh issue new` (= alias) 含む regex で良いか**: 上記 sketch は `create|new` 両対応、kawaz の好みで縮小可
7. **rename タイミング**: 仕様確定後 `for-me/rules/gh-issue-kawaz-repo-guard.md` に rename、`.draft-` prefix を外す
